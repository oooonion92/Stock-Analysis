# 深度行业研究通用方法论

本文档是 `04_industry_research` 的第一层通用流程。它用于把 Zotero 中的研报、文章、网页和快照，沉淀成可复用的行业知识体系，再输出给个股分析和每日复盘。

## 0. 定位与边界

研究链条：

```text
Zotero 资料收集 -> 深度行业研究 -> 个股分析 -> 每日复盘/交易预案
```

本子项目只保存结构化成果和必要索引，不把大量原始 PDF、HTML 快照或附件复制进仓库。原始资料仍由 Zotero 保存。

第一版优先服务投资框架，日常复盘和个股分析是下游输出，不抢主线。

## 0.1 增量更新原则

行业研究不是每拿到一篇新研报就重写一次底层逻辑。后续新增研报、新资料和新行业知识，默认在既有框架下增量吸收：

- 新资料先进入事实索引，标明来源、时间、证据、涉及环节和可疑点。
- 若新资料只是补充细节，加入对应环节、公司、变量或催化剂，不改核心框架。
- 若新资料提供新的公司映射，先加入公司清单或待验证问题，不直接升格为核心公司。
- 若新资料提供新的风险或反证，优先加入“风险与反证”和“跟踪指标”。
- 只有当多条高质量证据持续推翻原核心矛盾、产业链传导或公司受益路径时，才更新底层假设。
- 底层假设发生修改时，必须保留修改原因、证据来源和修改日期，避免研究框架随市场情绪频繁漂移。

推荐把更新分成三类：

| 更新类型 | 处理方式 |
| --- | --- |
| 事实补充 | 加入事实索引、资料索引、对应环节备注。 |
| 框架增强 | 在原章节下新增变量、公司、催化剂或反证。 |
| 框架修正 | 明确写出旧判断、新判断、触发证据和仍需验证的问题。 |

## 1. 资料入口

每次开始行业研究时，先确认资料入口：

| 字段 | 要求 |
| --- | --- |
| 行业名称 | 使用稳定名称，例如 `PCB`、`机器人`、`光模块` |
| Zotero collection/tag | 记录 collection 名、collection key、tag |
| 资料范围 | 说明是否只读 collection，还是叠加 tag、网页搜索、公司公告 |
| 附件状态 | 标记 HTML 快照、PDF、无附件、无法读取 |
| 时间范围 | 记录资料日期或访问日期，避免过期资料混入最新判断 |

资料索引使用统一格式：

```markdown
| Zotero key | 标题 | 来源URL | 日期/访问日期 | 附件状态 | 主题标签 | 关键结论 |
| --- | --- | --- | --- | --- | --- | --- |
```

附件状态建议使用：

- `html_snapshot_readable`: 有本地 HTML 快照且正文可读。
- `pdf_link_only`: 只有 PDF 链接，需要单独下载或读取。
- `pdf_storage_readable`: Zotero API 未显示附件，但 `C:\Users\翀\Zotero\storage\<Zotero key>\` 下存在可读 PDF。
- `web_link_only`: 只有网页链接，正文未保存。
- `blocked_or_paywalled`: 当前只能看到登录、注册或付费墙。
- `needs_manual_review`: 抽取结果异常，需要人工核对。

如果 Zotero API 条目没有返回子附件，但条目是 PDF 或网页保存失败，必须再检查本地目录：

```text
C:\Users\翀\Zotero\storage\<Zotero key>
```

该目录下可能存在 `.pdf`、`.html`、`.zotero-ft-cache` 或阅读器状态文件。若 PDF 路径包含中文用户名导致脚本读取失败，可复制到临时 ASCII 路径读取，但不要复制进项目仓库。

## 2. 事实索引

每条资料先做事实索引，再进入观点归纳。不要直接把文章观点当成行业结论。

每条资料至少提炼：

| 字段 | 说明 |
| --- | --- |
| 核心观点 | 文章真正想证明什么 |
| 关键证据 | 数字、事件、调研反馈、价格、产能、订单、客户认证 |
| 产业链环节 | 上游材料、中游制造、下游应用、设备耗材、渠道/客户 |
| 涉及公司 | 公司名称和对应受益/受损环节 |
| 可疑点 | 数据口径不清、转述链条过长、情绪化判断、过度线性外推 |
| 可复用程度 | 高、中、低；决定是否进入行业框架 |

事实索引的原则：

- 来源转述要标明来源层级：一手公告、券商研报、媒体转述、调研纪要、市场传闻。
- 数字不孤立使用，必须记录口径和时间。
- 公司不只列名称，必须说明对应环节和受益路径。
- 对没有证据支撑的结论，先放入 `待验证问题`。

## 3. 行业框架

行业框架按以下顺序搭建。先拆核心矛盾，找到行业真正的卡点；再沿上游、中游、下游产业链展开，判断卡点如何传导、利润如何分配、哪些公司真正受益。

1. 核心矛盾：行业现在最关键的供需、技术、价格或竞争矛盾是什么。
2. 核心卡点：需求、供给、价格、库存、技术路线、客户认证、政策或竞争格局中哪一项最限制行业。
3. 上游拆解：原材料、关键零部件、设备耗材、核心技术和供给瓶颈。
4. 中游拆解：制造、集成、加工、良率、产能、成本结构和议价权。
5. 下游拆解：终端需求、客户结构、应用场景、订单持续性和替代路线。
6. 产业链传导：下游需求如何向上游传导，上游瓶颈如何反向约束中下游。
7. 投资落点：哪个环节最先受益，哪个环节弹性最大，哪个环节确定性最高。

### 3.1 从行业故事到稀缺层

吸收 `serenity-skill` 的供应链瓶颈研究思路后，行业框架增加一条硬约束：先排产业链层级，再排公司。不要从热门股票或市场故事开始，而要先回答：

```text
市场故事 -> 系统变化 -> 必要零部件/工艺 -> 产业链层级 -> 稀缺层 -> 公司映射 -> 证据 -> 反证
```

研究时先写清：

| 步骤 | 要回答的问题 |
| --- | --- |
| 市场故事 | 市场现在在交易什么叙事？ |
| 系统变化 | 真实的技术、经济或客户结构发生了什么变化？ |
| 约束变量 | 哪个物理或经营约束被放大：功耗、带宽、良率、纯度、可靠性、产能、交期、认证、资本开支？ |
| 产业链层级 | 这个约束落在上游材料、设备耗材、中游制造、下游客户还是基础设施？ |
| 稀缺层 | 哪一层最难扩、供应商最少、认证最慢、客户最难绕开？ |
| 公司映射 | 哪些公司控制稀缺层，哪些只是供应稀缺层，哪些只是主题受益？ |
| 反证条件 | 什么事实出现，说明这个稀缺层判断错了？ |

稀缺层排序优先于公司排序。一个公司即使涨得强、叙事热，如果没有贴近稀缺层或证据不足，也只能列为线索或映射受益。

稀缺层判断的常见信号：

- 供应商数量少，替代验证慢。
- 扩产需要专用设备、工艺 know-how、材料纯度、客户认证或长周期资本开支。
- 客户出现预付款、长单、产能锁定、交期拉长、涨价接受或加急订单。
- 公司仍被市场按旧业务定价，但收入结构、毛利率或客户结构正在向新稀缺层迁移。
- 下游需求很热，但真正限制交付的是上游材料、设备、良率、认证或基础设施。

行业结论使用统一格式：

```markdown
| 结论 | 证据 | 受益环节 | 相关公司 | 反证/风险 | 跟踪指标 |
| --- | --- | --- | --- | --- | --- |
```

## 3.2 证据等级

行业研究必须区分事实强度。新增资料不能因为叙事顺耳就进入核心结论。

| 证据等级 | 来源类型 | 使用方式 |
| --- | --- | --- |
| 强证据 | 年报、季报、临时公告、交易所文件、问询函回复、官方订单/中标/合同、客户认证、监管/项目备案、专利/标准/技术论文 | 可支撑核心结论和公司排序。 |
| 中证据 | 可信媒体、行业协会、贸易出版物、公司官网/产品页、假设透明的券商研报、跨公司公开信息交叉验证 | 可支持判断，但关键结论需继续找强证据。 |
| 弱证据 | KOL、社媒、论坛、来源不明截图、未署名渠道反馈、单纯股价异动 | 只能作为线索，不能单独支撑核心公司或高置信结论。 |

进入核心公司清单的公司，至少要回答：

- 它到底卡住或贴近哪个环节？
- 有什么证据说明需求、产能、客户、认证、收入或毛利正在变化？
- 当前证据是强、中、弱哪一级？
- 还缺什么关键证明？
- 什么情况出现应当下调判断？

## 4. 投资翻译

行业研究最终要翻译成可跟踪的投资框架，而不是停在知识介绍。

输出必须回答：

- 哪个环节最先受益，哪个环节弹性最大，哪个环节确定性最高。
- 哪些产业链稀缺层最值得优先研究，为什么排在前面。
- 哪些公司是直接受益，哪些只是映射或情绪联动。
- 催化剂是什么：业绩、订单、涨价、扩产、客户认证、政策、技术节点。
- 反证条件是什么：需求下修、扩产过快、价格回落、良率不达预期、客户认证失败。
- 跟踪指标是什么：价格、交期、订单能见度、产能利用率、库存、资本开支、客户结构。

不要把“行业好”直接等同于“股票可以买”。行业研究只解决产业逻辑和公司映射，交易决策仍需要进入个股分析和每日复盘流程。

## 5. 下游输出

### 输出给个股分析

为 `03_stock_deep_analysis` 提供：

- 行业核心矛盾。
- 公司所处产业链环节。
- 受益路径和利润弹性来源。
- 行业催化剂与公司催化剂的区别。
- 公司反证条件和需要跟踪的经营指标。

### 输出给每日复盘

为 `02_daily_replay` 提供：

- 当日热点是否有产业逻辑支撑。
- 催化剂是短期事件还是中期主线。
- 主线环节排序：下游需求、中游制造、上游材料、设备耗材。
- 当日异动公司属于核心受益、映射受益还是边缘情绪。
- 需要进入后续跟踪的价格、订单、产能、政策或业绩事件。

## 6. 研究报告成稿逻辑

行业研究的最终报告要像一份可阅读的研究稿，而不是工作过程记录。正文优先放研究判断、证据链、产业链拆解、公司拆解和反证；资料来源、工具选择、制作说明等内容默认留在事实索引、研究计划或附录文件，不放进 HTML 正文，除非用户明确要求展示。

推荐正文顺序：

1. 一句话主结论或核心判断。
2. 核心矛盾：先说明本轮行业变化的本质和卡点。
3. 稀缺层排序：先排产业链层级，再排公司。
4. 关键结构图或机制图：帮助读者建立第一性理解。
5. 上游 -> 中游 -> 下游产业链图谱。
6. 关键变量和卡点：需求、供给、价格、产能、良率、技术路线、认证。
7. 上游/中游/下游细分环节拆解。
8. 核心公司深度拆解：业务强势方向、技术壁垒、经营状况、受益路径、证据等级、反证和跟踪指标。
9. 催化剂、风险与反证。
10. 下游同步摘要：给个股分析和每日复盘使用。

报告正文的删减原则：

- 删除“工具怎么选”“怎么制作图片”“资料来源列表”等制作痕迹。
- 删除对读者没有直接研究价值的元信息标签。
- 保留必要的图注和 caveat，说明图是结构示意还是精确产品图。
- 来源信息必须在事实索引或研究计划中留存；正文可以只保留关键出处或脚注式引用。

## 7. 可视化与图稿流程

行业图稿先服务研究问题，不先追求好看。每张图开始前先定义它要回答什么问题：

| 图稿类型 | 适合回答的问题 | 注意事项 |
| --- | --- | --- |
| 结构/剖面图 | 产品结构、材料位置、工艺难点 | 先保证结构正确；写实底图可用生成图，但中文标注建议后期可控叠加。 |
| 产业链图谱 | 上游、中游、下游和利润传导 | 按产业链顺序组织，不把需求端热度直接等同于公司受益。 |
| 变量传导图 | 需求、价格、产能、库存、利润率如何传导 | 保留反证路径，不只画单向利好。 |
| 公司分层图 | 核心受益、弹性受益、映射受益 | 每家公司必须绑定受益环节和反证。 |

图稿生成的推荐流程：

1. 先写清图的任务：解释结构、解释传导、解释公司分层，三者不要混在一张图里。
2. 如果需要写实质感，先生成无字底图，避免生成模型把中文标签画错。
3. 中文标签、箭头、编号和图例优先用 HTML/SVG/Canvas 后期叠加，保证可修改、可校对。
4. 生成图必须落地为项目资产，例如 `assets/<industry>_*.png`，不要只停留在临时生成目录。
5. 报告引用 PNG 或 HTML 内嵌图，避免使用可点击 SVG 作为入口。
6. 每张图完成后做截图验收：检查文字、箭头、遮挡、裁切、移动端缩放和文件是否可打开。

可用工具定位：

| 工具/插件 | 更适合做什么 | 边界 |
| --- | --- | --- |
| Build Web Data Visualization | 可控信息图、产业链图、HTML 报告、截图导出 | 适合确定性强的图表和中文标注。 |
| Image generation | 写实底图、材质感插图、封面级视觉 | 不适合直接负责精确中文标注和数据表达。 |
| Product Design | 多个视觉方向探索、报告视觉系统选择 | 适合先看方向，再选一个落地。 |
| Creative Production | moodboard、风格关键词、视觉语言选择 | 适合定审美路线，不替代研究判断。 |
| Spreadsheets | 公司清单、变量跟踪、价格/交期/业绩表 | 适合沉淀结构化跟踪，不替代正文研究。 |

## 9. 质量检查

完成一份行业研究前，至少检查：

- 每个核心结论是否有来源索引。
- 每个受益逻辑是否有对应反证。
- 每个核心公司是否说明了产业链环节和受益路径。
- 是否区分事实、推论和待验证问题。
- 是否能输出给个股分析和每日复盘，而不只是行业介绍。
- 正文是否删除了制作过程、工具选择和冗长资料列表。
- 关键图稿是否经过视觉验收，且没有错误材料映射、文字重叠或误导性比例。
